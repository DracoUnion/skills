# find-space-through-external-perspective

## When to Use

当个人或项目陷入困境、感觉无路可走或视角受限时，用于寻找新的解决方案和发展空间description

## Core Logic

# 执行步骤

1.  **问题维度评估与目标拆解**
    *   **输入**：一个模糊或复杂的未知问题/任务/目标。
    *   **操作**：
        a. **评估维度**：判断问题是“二维问题”（因果关系直接、变量少）还是“三维以上问题”（涉及多个利益相关方、动态反馈、长期影响、生态复杂）。
        b. **拆解目标**：将总目标分解为若干个具体、可执行或可搜索的关键问题或子任务模块。
    *   **判断标准**：能够清晰陈述问题复杂度，并生成一份具体、可行动的子问题/任务清单。
    *   **IF** 判断为“二维问题” -> 采用“可见层思维”（直接、高效地解决表面问题），流程结束。
    *   **ELSE** (判断为“三维以上问题”) -> 进入步骤2。

2.  **识别不可见层结构与能力缺口**
    *   **输入**：步骤1中识别的复杂问题及拆解后的子任务清单。
    *   **操作**：
        a. **分析系统结构**：识别问题的底层系统要素，包括：**趋势与模式**（行业/技术发展）、**生态关系**（各参与方的共生/竞争）、**风险与对冲**可能性、**资源创造与分配逻辑**（零和/正和）。
        b. **分析能力缺口**：针对每个子任务模块，评估自身/团队是否具备独立完成所需的专业知识、技能、时间或资源。标记“可独立完成”或“存在能力/资源缺口”。
    *   **判断标准**：能够描述系统中至少三个关键要素的动态关系，并生成一份带有“缺口”标记的任务清单。
    *   **IF** 所有任务均可独立完成 -> 进入个人执行流程。
    *   **ELSE** (存在一个或多个缺口) -> 进入步骤3。

3.  **设计系统级干预与外部资源连接策略**
    *   **输入**：步骤2输出的系统结构洞察与带缺口标记的任务清单。
    *   **操作**：
        a. **制定系统干预策略**：基于系统洞察，设计能作用于底层结构的干预点，策略方向包括：**顺应趋势**（借鉴先进模式）、**强化生态**（促进行业健康）、**对冲风险**、**设计正和游戏**（激励共同创造）。
        b. **识别外部资源**：针对每个“缺口任务”，识别可弥补该缺口的资源类型，包括：专业人士（专家、顾问）、专业工具/平台、成熟产品/服务、协作网络（供应商、合作伙伴）、用户群体（UGC潜力）。
        c. **评估与选择整合方案**：评估各备选资源的成本、可靠性、兼容性，为每个任务选择最优整合方案（如购买服务、委托合作、使用工具、咨询专家、激发UGC）。
    *   **判断标准**：形成一份“系统干预与资源整合决策表”，明确系统杠杆点及每个任务将由何种外部资源以何种方式完成。
    *   **常见误区**：只关注眼前具体矛盾，忽视系统趋势；试图寻找“完美”单一资源；仅考虑直接成本，忽略整合隐性成本。

4.  **建立协作机制与执行验证**
    *   **输入**：“系统干预与资源整合决策表”。
    *   **操作**：
        a. **建立调用机制**：根据所选方案，建立具体的调用或协作流程，明确关键接口（联系人、API、交付标准、沟通机制）。
        b. **执行双线计划**：
            *   **设计主任务线 (Plan A)**：在理想/最可能假设下，制定完整行动方案。
            *   **设计备选任务线 (Plan B)**：针对2-3个关键障碍（如关键资源失效、政策变动），预设触发条件及应对方案（方法调整、资源替代、路径迂回）。
        c. **聚焦核心与监控**：将个人精力聚焦于无法外包的核心战略、创意整合或关系管理。监控外部资源交付与整体进展，根据预设信号在Plan A与Plan B间有序切换。
    *   **判断标准**：外部资源处于“就绪”状态；整体计划（A+B）逻辑协调且有弹性；个人时间向高价值活动倾斜。
    *   **常见误区**：微观管理已分包任务；对外部资源完全放任；对障碍触发信号视而不见或过早切换。

5.  **视角切换、迭代与认知升级**
    *   **输入**：执行中的项目、来自外部资源的反馈、用户/市场反应。
    *   **操作**：
        a. **强制性视角切换**：当决策选项有限或陷入僵局时，主动切换至**外部角色视角**（如全新决策者）、**极端用户视角**、**跨行业类比视角**或**时间视角**，生成新选项。
        b. **沉浸式观察（三行而后思）**：当过度分析导致停滞时，进入真实用户场景进行沉浸式观察与感受，基于具体感受发起小规模验证行动，再进行分析迭代。
        c. **问题驱动学习**：将执行中遇到的具体问题作为学习锚点，通过探索验证获得新知，迭代个人认知体系中的“旧识”。
        d. **动态替换与重构**：若方案部分失效或环境巨变，返回步骤3，针对失效模块重新搜索、评估并替换，进行局部或全局重构。
    *   **判断标准**：能够基于新视角或真实反馈产生创新方案；学习循环围绕具体问题完成认知交换；方案具备动态适应性。
    *   **常见误区**：固执于初始方案；切换视角流于表面；学习脱离具体问题。

# 输出格式要求

输出一份结构化的“系统化解决方案与资源杠杆报告”，包含：
1.  **核心问题与维度判断**：陈述原始问题，并说明其为何是“三维以上问题”。
2.  **系统结构洞察与任务缺口**：
    *   **关键系统要素**：趋势、生态关系、风险、资源逻辑。
    *   **任务分解与状态**：列出子任务及“可独立完成/存在缺口：具体说明”。
3.  **整合策略与双线计划**：
    *   **系统干预点**：建议的杠杆策略及理由。
    *   **外部资源整合方案**：针对每个缺口任务的推荐资源/方案及理由。
    *   **关键障碍与应对**：列出2-3个关键障碍、其触发信号及对应的Plan B简述。
4.  **执行状态与迭代方向**：
    *   **当前计划线**：正在执行Plan A / 已切换至Plan BX。
    *   **近期行动**：列出1-3项立即可以启动的、与调用资源或验证假设相关的具体行动。
    *   **待验证假设**：基于视角切换或观察感受提出的、待验证的新方向或认知点。

[CRITIC]
1.  **逻辑闭环缺陷**：步骤5的“动态替换与重构”分支指示“返回步骤3”，但步骤3的输入依赖于步骤2的“缺口分析”。若在项目后期进行重构，原始的任务缺口清单可能已过时，直接返回步骤3可能导致基于旧框架的决策。建议明确在迭代循环中，需要首先“重新评估当前目标与任务分解”，再进入资源识别与方案选择。
2.  **冲突条件未标注**：多个原始Skill对“是否寻求外部帮助”有不同判断分支。本合并版本以`resource-leveraging-framework`和`professional-gap-bridging`的权威逻辑为主（先评估缺口，再决定是否寻求外部资源），但未明确标注与其他Skill（如`find-space-through-external-perspective`默认需外部视角）在此处的逻辑取舍。
3.  **缺少回退方案**：在步骤4的“建立协作机制”中，仅提及“明确交付标准”，但未包含当外部资源交付不达标时的具体回退或补救措施（例如，合同中的违约金条款、备选服务商的快速切换流程）。这增加了执行风险。
