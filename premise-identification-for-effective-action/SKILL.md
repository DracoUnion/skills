# premise-identification-for-effective-action

## When to Use

当接收到一个明确的方法、指令或答案，准备执行前，调用此模型检验其有效性前提。description

## Core Logic

# 执行步骤

1.  **提取方法陈述**：明确记录下你获得的或你计划采用的特定方法、指令或“答案”。
    *   **输入**：来自手册、建议、经验或自己想法的具体操作描述（例如：“拉阻风门到底以启动摩托车”）。
    *   **输出**：一句清晰的方法陈述句。

2.  **逆向追问前提条件**：对该方法陈述进行系统性前提追问。
    *   **操作**：反复追问：“这个方法要成立并有效，必须满足什么前提条件或处于什么背景下？”至少追问三层。
        *   第一层：直接操作前提（例如：发动机是冷的）。
        *   第二层：环境或系统前提（例如：摩托车燃油系统正常，电池有电）。
        *   第三层：人或关系前提（例如：操作者理解阻风门的基本原理）。
    *   **输出**：一个分层次的前提条件清单。

3.  **验证前提匹配度**：将列出的前提条件与当前实际情况进行逐一比对。
    *   **输入**：前提条件清单、对当前实际情况的观察与数据。
    *   **操作**：判断每个前提条件在当前情境下是否成立。
    *   **输出**：一份匹配度报告，标明哪些前提**成立**，哪些**不成立**，哪些**不确定**。

4.  **做出执行决策**：
    *   **IF** 所有关键前提条件均成立：**THEN** 该方法可执行，按计划行动。
    *   **ELSE IF** 有任一关键前提不成立：**THEN** 该方法在当前情境下无效。转向解决缺失的前提（这可能成为新的核心阻力），或寻找其他适应现状的方法。
    *   **ELSE** (关键前提不确定)：**THEN** 暂停执行，先设计小实验或收集信息来验证该前提，避免盲目行动。

# 输出格式要求
以清单形式输出分析结论：
- **待检验方法**：[引用原方法]
- **关键前提条件**：
    1.  [前提条件1] - 匹配状态：[成立/不成立/不确定]
    2.  [前提条件2] - 匹配状态：[成立/不成立/不确定]
- **执行建议**：[基于匹配状态给出明确建议：执行/调整/暂停并验证]
